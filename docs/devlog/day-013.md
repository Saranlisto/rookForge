# Day 013 - Castling

## Summary

Added castling generation, castling legality checks, and castling move application. Castling is represented as normal king moves: `e1g1`, `e1c1`, `e8g8`, and `e8c8`.

## Implemented

- Added `generate_castling_moves(&Position) -> Vec<Move>`.
- Added castling moves to `generate_legal_moves`.
- Added castling application to `apply_move`.
- Added castling rook movement:
  - White kingside: king `e1g1`, rook `h1f1`
  - White queenside: king `e1c1`, rook `a1d1`
  - Black kingside: king `e8g8`, rook `h8f8`
  - Black queenside: king `e8c8`, rook `a8d8`

## Castling Rules Checked

- Required castling right must exist.
- King must be on its original square.
- Rook must be on its original square.
- Squares between king and rook must be empty.
- King must not currently be in check.
- King must not pass through an attacked square.
- King must not land on an attacked square.

## Tests Added

- White kingside and queenside castling generation
- Black kingside and queenside castling generation
- No castling without rights
- No castling through occupied paths
- No castling while in check
- No castling through attacked transit squares
- No castling onto attacked destination squares
- Applying all four castling directions moves the rook correctly
- Castling removes castling rights for that color
- Castling increments the halfmove clock
- Legal move output includes castling from a castling-ready position
- CLI integration coverage for castling legal moves

## Local Checks

The local smoke workflow now includes:

```bash
cargo run -- movegen legal --fen "r3k2r/8/8/8/8/8/8/R3K2R w KQkq - 0 1"
```

Run the complete validation workflow before committing:

```bash
make check
```
